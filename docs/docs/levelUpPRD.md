**Status**: Draft
**Author**: Alex (PM)
**Last Updated**: 2026-03-31
**Version**: 1.0
**Stakeholders**: 運營者本人（開發者 + 內容創作者 + PM）

## 1. 產品願景與目標

### 一句話願景

讓一個人透過 AI 驅動的自動化系統，同時高品質運營多個垂直社群帳號，將 80% 的重複苦力交給機器，把 100% 的精力保留給觀點輸出與粉絲互動。

### 核心問題

同時經營三個不同領域的社群帳號，面臨以下痛點：

1. **心智耗損（Context Switching）**：在 AI 自動化、PMP 職涯、足球策略三個截然不同的領域之間頻繁切換，導致產能崩潰。
2. **內容生產瓶頸**：每天需要找素材、讀新聞、翻譯、總結、畫圖表，80% 時間花在重複的苦力活上。
3. **跨平台維護地獄**：Meta Graph API 授權繁瑣、X API 限制多且貴、不同平台格式完全不同，一旦改版全部報錯。
4. **缺乏數據回饋**：無法系統性追蹤哪種主題的留存率最高，決策靠直覺而非數據。

### 目標用戶

運營者本人 -- 一位具備 Python 開發能力、PM 產品思維、且對 AI/PMP/足球三個領域有深厚理解的「一人團隊」。

> 引用 Austin Lau 案例的核心啟示：「對業務邏輯的理解」比「寫代碼的能力」更重要。非技術人員可以藉由 AI 建立超級槓桿，而有技術背景的人更能將這個槓桿做到極致。

### 成功指標（總覽）

| 指標 | 基線 | 目標（12 週後） |
|------|------|-----------------|
| 每日內容生產時間 | 預估 3+ 小時/天 | < 30 分鐘/天（審核 + 互動） |
| 三帳號穩定更新率 | 0（尚未啟動） | 每帳號至少 5 篇/週 |
| 內容庫存量 | 0 | 40+ 篇已審核待發佈 |
| 數據驅動決策能力 | 無結構化數據 | 每週可跑主題效益分析 |

## 2. 核心假設

本系統建立在以下假設之上，依驗證優先級排列：

### 必須優先驗證（P0）

| 假設 | 驗證方式 | 失敗影響 |
|------|----------|----------|
| AI 生成的草稿品質足夠高，人類只需微調 20% 即可發佈 | 產出 30 篇草稿，測量「直接核准率」與「平均修改時間」 | 若品質過低，系統淪為文字產生器而非生產力工具 |
| 三個獨立帳號的「左手餵右手」飛輪能有效降低心智耗損 | 實際運營 4 週，記錄每帳號的生產流暢度與痛苦指數 | 若飛輪不轉，三線作戰會比預期更快導致停更 |
| 每帳號一天 1 篇的頻率足以在演算法中建立帳號權重 | 觀察 4 週的自然觸及率趨勢 | 若頻率不足，需調整為隔日或降低帳號數量 |

### 需要驗證（P1）

| 假設 | 驗證方式 |
|------|----------|
| 「資訊策展 + 個人觀點」的內容模式比純原創更高效且互動率不低 | A/B 比較策展類 vs 原創類貼文的互動率 |
| 「人情味」（碎碎念、血淚史、Build in Public）是 AI 時代最強護城河 | 追蹤有/無人情味加持的貼文留言數差異 |
| Calendar Marketing（事件前 24 小時預發）能有效搶佔流量頂端 | 比較「賽前預測」vs「賽後總結」的觸及率 |

### 底層假設（P2，暫不驗證）

- 社群平台的 API 在 MVP 期間不會有重大破壞性變更。
- 運營者有足夠的持續動力維持至少 8 週的測試期。
- 三個領域（AI、PMP、足球）各自存在足夠大的目標受眾。

## 3. 三帳號定位詳述

### 設計原則

> 三個獨立帳號 > 一個多標籤帳號。單一帳號混發不同領域內容會導致演算法「精神分裂」、受眾互相干擾、帳號權重分散。每個帳號必須有純粹的標籤定位。

### 帳號 A：AI 自動化（兵工廠）

| 維度 | 內容 |
|------|------|
| **定位** | 公開打造（Build in Public） |
| **目標受眾** | 開發者、創客（Maker）、效率追求者 |
| **核心內容** | 程式碼拆解（圖文）、腳本執行畫面（短影音）、AI 工具實戰教學 |
| **差異化** | 不只教工具，展現「為了營運另外兩個帳號」所寫的爬蟲與 API 串接過程（吃自己的狗糧） |
| **飛輪效應** | 為帳號 B、C 寫的自動化腳本，本身就是帳號 A 的爆款素材 |
| **預期平台** | Threads、X |
| **發佈時段** | 中午 12:00-13:00（午休充電時段） |
| **頻率** | 每天 1-2 篇（中午教學 + 晚上可選的碎碎念） |
| **認證優先級** | 中（防止被演算法誤判為機器人） |
| **語氣** | 技術宅的幽默感、帶著被 API 報錯折磨的崩潰感 |

### 帳號 B：PMP 職涯（高淨值變現池）

| 維度 | 內容 |
|------|------|
| **定位** | 讓 PMP 框架落地的實戰派 PM |
| **目標受眾** | 專案經理、職場中高階人士、考證族群 |
| **核心內容** | 框架圖解、商業新聞點評（圖文）、利害關係人管理實戰 |
| **差異化** | 不教死板考證理論，而是將「一人管理三帳號」的排程與推動複雜專案的經驗，轉化為敏捷開發與 PMP 框架的實戰案例 |
| **飛輪效應** | 一人運營三帳號的專案管理經驗，直接成為 PMP 實戰教材 |
| **預期平台** | Threads、LinkedIn |
| **發佈時段** | 早上 07:30-08:30（通勤思考時段） |
| **頻率** | 每週 3-5 篇（重質不重量，一天最多 1 篇） |
| **認證優先級** | 最高（專業防線 + 商業信任度） |
| **語氣** | 專業但帶點無奈的幽默感、實戰血淚史的共鳴感 |

### 帳號 C：足球英文（流量沙盒）

| 維度 | 內容 |
|------|------|
| **定位** | 教育娛樂化（Edutainment）-- 用足球教英文 |
| **目標受眾** | 硬核球迷、戰術愛好者、想透過興趣學英文的人 |
| **核心內容** | 數據圖表、賽後重點整理（圖文）、賽前戰術預測、體育英文術語拆解 |
| **差異化** | 結合 Python 數據分析（matplotlib 球員跑動熱區圖、傳球網路圖）+ 英文情境學習，不當「老師」而是「陪練員」 |
| **飛輪效應** | 用 Python 抓取的足球數據和自動繪製的圖表，同時作為帳號 A 的技術火力展示 |
| **預期平台** | Threads、X、IG |
| **發佈時段** | 晚上 19:30-21:00（下班娛樂時段） |
| **頻率** | 非比賽日 1 篇 / 比賽日 2-3 篇（賽前預測、半場快評、賽後數據） |
| **認證優先級** | 低（球迷更看重數據準不準，但藍勾勾能搶到留言區頂級版位） |
| **語氣** | 資深球迷的熱情、帶有個人實戰觀賽心得的真實感 |

### 內容飛輪機制

```
技術實現（為帳號 B、C 做自動化）
  --> 完整記錄過程（代碼、遇到的坑）
  --> 轉換成帳號 A 的爆款內容
  --> 帳號 B 提煉管理框架
  --> 帳號 C 搭順風車用數據分析
  --> 三帳號互相引流、互相驗證
```

## 4. 六大模組需求概述

### 模組 1：內容策展大腦（Curation Brain）

**目的**：取代人工找素材、讀新聞、翻譯、總結的 80% 苦力活，讓 AI 每天自動備好草稿。

**核心功能**：

1. **資料抓取（Extract）**
   - Python 爬蟲定時抓取目標來源（RSS Feed、Football-Data.org API、科技商業新聞網站）
   - 針對三帳號分別鎖定不同資料源
   - 支援 Hugging Face Trending Models、GitHub Trending Repos 等科技日曆

2. **AI 分析與評分（Transform）**
   - 透過 Anthropic/Gemini API，使用結構化 Prompt 處理原始資料
   - 強制 AI 輸出 JSON 格式（標題、內文、重點標籤、Reasoning 欄位）
   - 三套不同的 System Prompt 對應三個帳號風格
   - AI Skill #1：爆款內容拆解（分析 Hook、結構、情緒觸發點）
   - AI Skill #2：趨勢預測（判斷哪些會成為流量熱點）
   - AI Skill #3：粉絲評論分析（評分留言質量、總結用戶痛點）

3. **結構化存儲（Load）**
   - SQLite 本地數據庫（不用架 Server）
   - 結構化欄位：ID、發布日期、素材來源、內容屬性標籤、核心內容、Reasoning、人情味備註、績效數據、Account_Type
   - 自動生成 Matplotlib 圖表（足球戰術用）
   - Markdown 備份至 GitHub Private Repo 或 Obsidian

**依賴關係**：
- 下游：HITL 審核系統（輸出 DRAFT 狀態內容）
- 下游：先知模組（共用抓取基礎設施）
- 上游：數據分析迴圈（回饋優化 Prompt 權重）

---

### 模組 2：先知模組（Oracle Engine）

**目的**：從「追逐流量的跟風者」升級為「佈局流量的造浪者」，在事件發生前 24 小時發布預測/科普，搶佔流量頂端。

**核心功能**：

1. **行事曆抓取**
   - 體育賽事：API-Football 抓取未來 7 天賽程（FIFA、英超、歐冠）
   - 科技發布：監控 Apple WWDC、Google I/O、OpenAI DevDay 等已知日曆
   - 商業週期：季末績效考核、聯準會利率決策、大型科技股財報週

2. **AI 視角生成**
   - 足球帳號：賽前戰術預測、反直覺數據亮點、球迷討論問題
   - AI 帳號：新技術對工作流的影響預測
   - PMP 帳號：大公司決策的 PMP 框架啟示、季末向上管理短文

3. **預排程系統**
   - 內容標記為 PREDICTION 類型
   - target_event_date 欄位追蹤目標事件日期
   - 自動排程在事件前 12-24 小時發布

4. **雙軌生成引擎**
   - 回顧引擎：抓昨天的資料，寫今天的總結（NEWS_RECAP）
   - 預測引擎：抓下週的行事曆，寫下週的預熱草稿（PREDICTION）

**依賴關係**：
- 上游：內容策展大腦（共用抓取與 AI 處理基礎設施）
- 下游：HITL 審核系統（預測內容需要更嚴格的人工校驗）
- 下游：數據分析迴圈（比較預測 vs 回顧內容的觸及率）

---

### 模組 3：HITL 審核系統（Human-in-the-Loop）

**目的**：確保 AI 不會一本正經胡說八道，注入人情味，保持品牌一致性，隨時可以暫停異常內容。

**核心功能**：

1. **狀態機設計**
   ```
   DRAFT (草稿)
     --> [AI 生成]
   PENDING_REVIEW (待審)
     --> [你判斷 & 改]
   APPROVED (已核准)
     --> [自動發布]
   PUBLISHED (已發布)
     --> [監控互動]
   ANALYZED (分析完成)
   REJECTED (已捨棄)
   ```

2. **審核介面（二選一）**
   - 方案 A（終端機互動）：`audit.py` 一條條跳出 DRAFT 內容，支援 (A) 核准 / (E) 修改並核准 / (D) 捨棄
   - 方案 B（Markdown 檔案）：匯出 `review.md`，在 VS Code 或手機筆記軟體編輯後回寫資料庫

3. **批次審核機制**
   - 週六 AI 腳本一次生成下週 21 篇 DRAFT
   - 週日晚上 15 分鐘批次審核
   - 核准後整週自動分發，不需再操心

4. **預檢系統（Pre-flight Check）**
   - 自動驗證字數是否符合目標平台限制
   - 檢查是否有必要的附圖（IG 強制需要圖片）
   - 圖片是否已上傳至雲端取得公開 URL

**依賴關係**：
- 上游：內容策展大腦 + 先知模組（接收 DRAFT 內容）
- 下游：跨平台發佈系統（輸出 APPROVED 內容）
- 核心原則：防止 AI 幻覺、時事校正、語氣微調、每篇都必須說明「為什麼會火」

---

### 模組 4：跨平台發佈系統（Publishing Pipeline）

**目的**：統一 API 層，消滅為每個平台分別寫串接邏輯的噩夢，實現一份母體內容自動適配多平台。

**核心功能**：

1. **多態轉換器（Platform Adapter）**
   ```
   母體內容（統一格式）
     --> Adapter 識別目標平台
       --> Threads 轉換器（超過 500 字自動拆分串文）
       --> X 轉換器（超過 280 字截斷或分段）
       --> IG 轉換器（無圖片則擋下，標示 MediaRequired）
       --> LinkedIn 轉換器（長文格式適配）
   ```

2. **媒體中轉站（Media Staging Area）**
   - Matplotlib 圖表存為本地 PNG
   - 自動上傳至 Imgur API 或 Cloudflare R2（免費額度）
   - 取得公開 URL 後呼叫平台 API
   - 發文成功後可選擇刪除雲端暫存圖片

3. **排程分發器（Dispatcher）**
   - Python 腳本 + macOS launchd 每小時觸發
   - 掃描 SQLite 中 APPROVED + scheduled_time <= 現在時間的內容
   - 精準打擊黃金時段（早 07:30 PMP / 中 12:30 AI / 晚 20:00 足球）

4. **錯誤重試與狀態追蹤**
   - RATE_LIMIT_EXCEEDED：寫入 retry_after 時間戳，下一小時重試
   - PARTIAL_SUCCESS：記錄「Threads 已發、X 待補發」，避免重複貼文
   - API 超限自動重試、帳號異常告警通知、發文失敗自動回滾

**依賴關係**：
- 上游：HITL 審核系統（接收 APPROVED 內容）
- 下游：自動化互動引擎（發佈成功後觸發互動監聽）
- 下游：數據分析迴圈（發佈後開始收集績效數據）

---

### 模組 5：自動化互動引擎（Engagement Engine）

**目的**：讓帳號從「只會發佈」升級為「會社交」，自動化處理高 ROI 的互動行為，同時避免被判定為 Spam。

**核心功能**：

1. **智能回覆（Inbound Reply）**
   - `check_my_comments.py` 定時抓取最近 3 篇貼文的新留言
   - AI 判斷留言意圖：提問 / 稱讚 / 酸民
   - 提問：從知識庫找答案，草擬專業回覆
   - 稱讚：幽默有人情味的致謝
   - 酸民：無視或高 EQ 回擊
   - 草稿存入 `pending_replies.csv`，人工確認後才發送

2. **自動導流（Outreach Commenting）**
   - 清單化 30 個目標領域大 V（每帳號 10 個）
   - 每 15 分鐘檢查大 V 是否發了新內容
   - AI 結合知識庫草擬「提供補充價值」的深度留言（非罐頭訊息）
   - 審核後手動發送，搶佔留言區前排

3. **轉發者獎勵系統（Repost Rewarder）**
   - 監聽轉發清單，自動點讚轉發者
   - SQLite `engagement_log` 去重，避免重複點讚
   - 大 V 轉發時通知運營者親自回覆

4. **優秀回覆篩選器（Comment Ranker）**
   - 每小時抓取新留言，AI 評分（1-10 分）
   - >= 8 分：自動點讚
   - 10 分（極度優秀）：通知運營者親自回覆
   - 無意義留言（純 emoji）：自動點愛心
   - 惡意留言 / Spam：自動隱藏或刪除

5. **防封號策略（Anti-Ban）**
   - 隨機延遲：`time.sleep(random.uniform(60, 300))`
   - 每日限額：每帳號主動留言不超過 20-30 則
   - 指紋模擬：User-Agent 模擬真實瀏覽器環境
   - 自動點讚速度控制：分開在 10 分鐘內完成

**依賴關係**：
- 上游：跨平台發佈系統（發佈成功後觸發監聽）
- 雙向：HITL 審核系統（回覆草稿需人工確認）
- 下游：數據分析迴圈（互動數據回流分析）

### 模組 6：數據分析迴圈（Analytics Feedback Loop）

**目的**：將三個月的累積數據轉化為調整方向的最強依據，形成「發佈 -> 收集 -> 分析 -> 優化」的正反饋閉環。

**核心功能**：

1. **績效數據收集**
   - 每週手動打點：按讚、留言、轉發、收藏、網站導流點擊數
   - 網站端追蹤：Google Analytics 或 Umami（開源輕量級）
   - 自動截圖備份（Playwright/Selenium，進階功能）

2. **數據分析儀表板（Python + Pandas + Matplotlib）**
   - 主題效益分析：哪個主題的收藏率最高？
   - 人情味相關性：有/無人情味加持的貼文留言數差異
   - 單字難度對比：大眾對多益單字 vs 地道運動術語的興趣差異
   - 轉化率漏斗：哪類內容最能導流到自建網站
   - 發文時段分析：不同時段的成效比較

3. **AI Prompt 優化回饋**
   - 將績效數據回饋至策展大腦
   - 根據數據調整下月 AI Prompt 的權重與偏好
   - 優化「爆款預測」的準確度

4. **淘汰機制（Kill Switch）**
   - 8 週後嚴格比較三帳號的「粉絲增長率」與「生產痛苦指數」
   - 數據最差或最痛苦的帳號果斷砍掉或降頻

**依賴關係**：
- 上游：所有其他模組（收集全鏈路數據）
- 下游：內容策展大腦（回饋優化 Prompt）
- 下游：先知模組（優化預測準度）
- 核心產出：三個月數據 = 訓練專屬 AI 模型（Fine-tuning）的最佳語料

## 6. MVP 範圍定義

### 階段劃分（共 12 週）

#### 第一階段：基礎建設（週 1-2）

**In Scope**:
- 決定三個帳號的最終定位與平台
- 設計並建立 SQLite 數據庫結構（含 Account_Type、status 狀態機、content_type 區分）
- 建立本地內容管理系統基礎架構

**Out of Scope**:
- 前端管理後台介面
- 自動截圖備份

#### 第二階段：爬蟲與策展（週 3-4）

**In Scope**:
- Python 爬蟲腳本：足球數據 API + 科技商業 RSS
- 3 套 AI Skills 的 Prompt 設計（拆解、預測、分析）
- 產出第一批 20-30 篇草稿
- 每篇草稿附 Reasoning 說明
- HITL 審核流程（終端機互動式 `audit.py`）
- 內容必須可編輯

**Out of Scope**:
- 先知模組的行事曆抓取
- Markdown 審核方案（先用終端機方案）

#### 第三階段：自動化發佈（週 5-6）

**In Scope**:
- Platform Adapter（Threads + X 轉換器）
- 媒體中轉站（Cloudflare R2 或 Imgur）
- `dispatcher.py` + macOS launchd 定時執行
- 基礎錯誤重試機制（RATE_LIMIT_EXCEEDED、PARTIAL_SUCCESS）
- 監控與日誌系統

**Out of Scope**:
- IG 轉換器（圖文製作流程尚未標準化）
- LinkedIn 轉換器
- YouTube 轉換器

#### 第四階段：互動引擎（週 7-8）

**In Scope**:
- 自動回覆模組（AI 分類 + 草稿生成 + 人工確認）
- 轉發者獎勵系統（自動點讚）
- 防封號策略（隨機延遲、每日限額）
- 8 週淘汰評估

**Out of Scope**:
- 自動導流模組（大 V 留言功能風險較高，延後）
- 優秀回覆篩選器的全自動模式
- ManyChat 等第三方對話自動化整合

#### 第五階段：數據分析（週 9-12）

**In Scope**:
- 收集與分析三個月的績效數據
- Python + Pandas 主題效益分析
- 評估三帳號表現，執行淘汰決策
- AI Prompt 權重回饋調整
- 先知模組 MVP（足球賽程抓取 + 賽前預測）

**Out of Scope**:
- 自建 React 管理後台
- Fine-tuning 專屬模型
- 付費工具引入（Make/Zapier 等）
- 短影音自動化生產
- 自建前端網站（React + TTS 語音播放器）

### MVP 核心原則

> **驗證假設 > 追求完美**。第 1 個月驗證「內容選題」是否對；第 2-3 個月累積數據、調整策略；第 4 個月開始才考慮擴大規模或引入付費工具。

## 7. 關鍵決策記錄

以下決策均提取自 Gemini 分析文件中的明確論述：

### 決策 1：三個獨立帳號 > 單一多標籤帳號

**決策**：為 AI、PMP、足球三個領域分別建立獨立帳號。

**原因**：
- 單一帳號混發會導致演算法「精神分裂」，無法進入任何垂直領域的推薦池
- 粉絲留存率互相衝突（球迷不想看 PMP 指標、PM 不關心越位戰術）
- 三帳號可作為 A/B/C 測試矩陣

**Trade-off**：心智耗損增加，需透過「左手餵右手」的內容飛輪機制緩解。

### 決策 2：每帳號一天 1 篇 > 一天 3 篇

**決策**：嚴格控制發文頻率，每帳號每天最多 1 篇（足球比賽日例外）。

**原因**：
- 一天 3 篇 x 3 帳號 = 9 篇 = 演算法懲罰
- 流量相食（Cannibalization）：第二篇會截斷第一篇的曝光
- 審核系統心智崩潰：63 篇/週的審核量不可接受
- 讓演算法有時間推廣，個別貼文互動率更高

**Trade-off**：總曝光次數可能較低，但單篇互動率與帳號權重更健康。

### 決策 3：捨棄 Make/Zapier，純代碼自建

**決策**：在 MVP 階段不使用 Make/Zapier 等第三方自動化平台。

**原因**：
- 運營者具備程式開發能力
- MVP 階段驗證假設優先於規模
- 可控性 > 方便性
- 純代碼方案可作為帳號 A 的「吃自己狗糧」爆款內容
- 節省月費

**Trade-off**：API 串接維護成本由自己承擔；但可借鑑 Boring Social Publisher 的架構精華（多態轉換器、媒體託管、錯誤重試）。

### 決策 4：圖文為主、短影音為輔

**決策**：MVP 階段以圖文/輪播圖為主力內容形式。

**原因**：
- 圖文適合拆解複雜邏輯、高資訊密度，收藏率高
- 生產門檻最低，降低「過度準備」導致的難產風險
- Text-First Approach：先測文字，數據好的主題再擴充為影音

**Trade-off**：放棄短影音的破圈能力和算法偏愛，但降低了生產阻力。

### 決策 5：Human-in-the-Loop 審核不可跳過

**決策**：所有 AI 生成內容必須經過人工審核才能進入發佈流程。

**原因**：
- 防止 AI 幻覺（比分記錯、術語解釋歪了）
- 注入人情味，保持品牌一致性
- 「完全不用動手」是社群毒藥，純機器人發文會被懲罰
- 「人情味」是 AI 時代最強護城河

**Trade-off**：每週需投入 15 分鐘審核時間，但這是不可削減的品質底線。

### 決策 6：資訊策展 + 個人觀點 > 純原創

**決策**：核心內容公式為「名人言論/熱點事件 + 你的獨特解讀」。

**原因**：
- 資訊策展的核心價值：替讀者省時間（解決 FOMO）
- 生產門檻極低：每天看新聞就等於在找素材
- AI 是最強輔助：自動抓取 RSS、AI 總結成 bullet points
- 公式：名人言論 + 你的解讀 = 雙重流量

**Trade-off**：同質性高，必須包含個人「毒舌評論」或「獨特視角」才能形成差異化。

### 決策 7：事件前 24 小時發 > 事件後才反應

**決策**：優先產出前瞻性預測內容，而非被動的事後總結。

**原因**：
- 等新聞出來才寫，只能分到長尾流量
- 事件前 24 小時發布，事件爆發時你已在頂部，自動成為權威聲音
- 足球賽事、科技發布會、財報季的時間都可預測

**Trade-off**：預測可能出錯，但錯誤預測本身也能成為互動話題（「打臉自己」的人情味）。

## 8. 風險與約束

### 技術風險

| 風險 | 可能性 | 影響 | 緩解措施 |
|------|--------|------|----------|
| AI 幻覺導致發佈錯誤資訊 | 高 | 高 | HITL 審核機制為硬性閘門，不可跳過 |
| 平台 API 突然改版或限制 | 中 | 高 | Platform Adapter 模式解耦；本地備份所有內容 |
| API 費用超預算 | 中 | 中 | 監控每月 API 呼叫量；設定每日限額 |
| launchd 排程失敗 | 低 | 中 | 加入失敗告警通知；錯過時段的內容自動進入下一輪 |
| SQLite 數據庫損壞 | 低 | 高 | Markdown 雙重備份至 GitHub Private Repo |

### 平台風險

| 風險 | 可能性 | 影響 | 緩解措施 |
|------|--------|------|----------|
| 帳號被演算法降權或 Shadowban | 中 | 高 | 防封號策略（隨機延遲、每日限額）；AI 帳號考慮認證以自證清白 |
| 帳號被封禁 | 低 | 極高 | 所有內容結構化備份於本地 SQLite + GitHub |
| Threads API 功能不完善 | 中 | 中 | API 剛開放不久，持續關注更新；備用手動發佈 |
| X API 費用過高 | 中 | 中 | Free 版限制多，評估 Basic 版（$100/月）的 ROI |

### 合規風險

| 風險 | 可能性 | 影響 | 緩解措施 |
|------|--------|------|----------|
| 版權侵權（使用體育賽事影像） | 中 | 高 | 圖文為主；影片使用綠幕/子母畫面的轉化性使用；使用平台原生 Remix 功能；優先使用 Creative Commons 素材 |
| 自動留言被判定為 Spam | 中 | 中 | 嚴格每日限額；「提供補充價值」的高品質留言而非罐頭訊息 |
| DMCA 下架 | 低 | 中 | 強調教育轉化性；使用公開片段（預告片、短片）；持續監測風險 |

### 人力約束

| 約束 | 影響 | 緩解措施 |
|------|------|----------|
| 一人團隊，心智耗損是最大敵人 | 高 | 左手餵右手飛輪；批量生產日制度；明確淘汰機制 |
| 上下文切換成本（三領域跨度大） | 高 | 分天處理（週六寫腳本/週日跑數據/週三寫 PMP） |
| 持續動力可能不足 | 中 | 8 週為硬性評估節點；痛苦指數過高的帳號立即降頻或砍掉 |

## 9. 成功指標（KPI）

### 第一階段：基礎驗證期（週 1-4）

| 指標 | 目標 | 衡量方式 |
|------|------|----------|
| AI 草稿「直接核准率」 | >= 50% | 審核時直接按 A 的比例 |
| 平均審核修改時間 | < 3 分鐘/篇 | 計時 |
| 第一批草稿產出量 | 20-30 篇 | SQLite 記錄 |
| 三帳號均已開始發佈 | 是/否 | 平台確認 |
| 每帳號穩定更新 | >= 3 篇/週 | 發佈記錄 |

### 第二階段：系統運轉期（週 5-8）

| 指標 | 目標 | 衡量方式 |
|------|------|----------|
| 內容庫存量 | >= 40 篇 APPROVED | SQLite 查詢 |
| 自動發佈成功率 | >= 95% | 發佈日誌 |
| 週日審核時間 | < 15 分鐘/21 篇 | 計時 |
| 每日運營總時間（含互動） | < 30 分鐘 | 自我記錄 |
| 發佈錯誤率（PARTIAL_SUCCESS） | < 5% | 狀態追蹤 |
| 自然觸及率趨勢 | 穩定或上升 | 平台後台 |
| 8 週淘汰評估完成 | 是/否 | 數據分析報告 |

### 第三階段：數據驅動期（週 9-12）

| 指標 | 目標 | 衡量方式 |
|------|------|----------|
| 單帳號粉絲增長 | >= 500 追蹤者（至少 1 個帳號） | 平台後台 |
| 平均互動率（Engagement Rate） | >= 3%（至少 1 個帳號） | 按讚+留言+轉發 / 觸及 |
| 收藏率 | 追蹤並比較 | 平台後台 |
| 預測性內容 vs 回顧性內容觸及率比較 | 預測 > 回顧 | 數據分析 |
| 「人情味」貼文 vs 純策展貼文留言數 | 人情味 > 純策展 | 數據分析 |
| 累積結構化數據量 | >= 200 篇含績效數據 | SQLite 記錄 |
| 是否決定擴大規模或引入付費工具 | 有明確決策 | 決策文件 |

### 北極星指標

**每週有效內容產出效率**：每分鐘投入時間所產出的「達到互動率目標」的貼文數量。

> 這個指標同時衡量了系統的自動化效率（分母越小越好）與內容品質（分子越大越好），是整個系統存在的根本理由。
