## 1. 產品發現 (Discovery Process)

### 1.1 用戶痛點分析

imgGen 解決的核心痛點是**內容創作者在「閱讀 → 視覺化 → 發布」流程中的高摩擦成本**。具體而言：

| 痛點 | 嚴重程度 | 頻率 | 現況成本 |
|------|---------|------|---------|
| 手動用 Canva/Figma 製作圖卡，每張耗時 10-20 分鐘 | 高 | 每日 | 每週 1-3 小時純排版工作 |
| 閱讀文章後需要另行整理重點再排版，認知切換成本高 | 高 | 每篇文章 | 從「閱讀模式」切換到「設計模式」的摩擦 |
| 現有工具缺乏 API 或 CLI，無法嵌入個人自動化工作流 | 中高 | 持續性 | 無法與 RSS、shell script、Notion 等工具串接 |
| 不同社群平台需要不同尺寸，手動調整耗時 | 中 | 每次發布 | 同一內容需重複調整 story/square/twitter 尺寸 |
| 多語言內容需求（zh-TW/en/ja 等），現有工具支援差 | 中 | 視內容而定 | 需手動翻譯或調整 |

**痛點量化估算**：以每天處理 5-15 篇文章、每週選 5-10 篇做圖卡的使用模式計算，手動方式每週耗費 1-3 小時在重複性排版上。若放棄圖卡直接發純文字，社群觸及率明顯下降。

### 1.2 市場機會

imgGen 切入的市場空間不是「設計工具」市場，而是**個人內容自動化管線**市場：

- **目標市場**: 獨立工作者、一人公司、知識型 KOL，擁有技術背景且熟悉 CLI
- **市場趨勢**: 社群平台演算法持續偏好圖文內容，純文字觸及率下降是長期趨勢
- **技術趨勢**: LLM API（Claude/Gemini/GPT）價格持續下降，使 AI 摘要的邊際成本趨近於零
- **未被滿足的需求**: 市面上沒有「從文章到圖卡到發布」的端到端 CLI 管線工具

imgGen 的定位不是與 Canva 競爭——它是「內容圖卡的 ffmpeg」：一個程式化、可腳本化、可嵌入自動化流程的管線工具。

### 1.3 競品分析

| 維度 | Canva | Figma | Carbon (程式碼截圖) | imgGen |
|------|-------|-------|---------------------|--------|
| **核心定位** | 通用設計平台 | 協作設計工具 | 程式碼美化截圖 | 文章→圖卡自動化管線 |
| **AI 摘要** | 無（需自行撰文） | 無 | 無 | 內建多 provider AI 摘要（Claude/Gemini/GPT） |
| **CLI / API** | 無原生 CLI | 無 | CLI 有限 | 完整 CLI + REST API + Web UI |
| **自動化能力** | 低（GUI 為主） | 低 | 中（有 CLI） | 高（batch、watch、preset、shell pipe） |
| **社群格式支援** | 手動選模板 | 手動調整 | 僅程式碼 | 4 種尺寸一鍵切換（story/square/landscape/twitter） |
| **模板數量** | 數千（通用） | 需自建 | 少數主題 | 28 種針對社群內容的專業主題 + article 模式 |
| **端到端管線** | 否 | 否 | 否 | 是：抓取 → AI 摘要 → 渲染 → 截圖 → 文案 → 發布 |
| **批次處理** | 付費功能 | 無 | 無 | 內建 async 並發批次（Semaphore 控制） |
| **成本模型** | 月費 $12.99+ | 月費 $12+ | 免費/付費 | 僅 AI API 調用費用（每張約 $0.001-0.01） |
| **多語言** | 依模板 | 手動 | N/A | 5 語言原生支援（zh-TW/zh-CN/en/ja/ko） |
| **技術門檻** | 低 | 中 | 低 | 中高（需熟悉 CLI/Python） |

**imgGen 的差異化護城河**：
1. **AI 原生**：不是「設計工具加上 AI」，而是「AI 摘要引擎加上設計渲染」——整條管線以 AI 為核心
2. **程式化**：可嵌入 shell script、cron job、RSS 自動化流程，這是 GUI 工具做不到的
3. **垂直專注**：不做通用設計，只做「文章內容 → 社群圖卡」這一件事，做到極致
4. **零 SaaS 依賴**：完全本地運行，數據不上雲，API key 自帶

### 1.4 目標用戶畫像

| 屬性 | 描述 |

| **日常行為** | 每天閱讀 5-15 篇英文或中文文章（科技、商業、文化類） |
| **社群習慣** | 定期在 Twitter/X、LinkedIn、Instagram 發布內容 |

| **最大挫折** | 花時間在設計排版上，而非內容本身 |
| **價值觀** | 重視效率、自動化、對工具的完全掌控；不願為 SaaS 月費付費 |

> **When** 我讀完一篇好文章、想把精華分享到社群時，
> **I want to** 快速將文章重點轉化為視覺化圖卡，

## 3. 問題陳述 (Problem Statement)

### 3.1 當前痛點的量化描述

**核心問題**: 社群媒體上純文字貼文的觸及率持續下降，內容創作者需要視覺化內容來維持影響力。但手動製作圖卡的成本與頻率需求之間存在結構性矛盾。

| 指標 | 量化數據 |
|------|---------|
| 每張圖卡手動製作時間 | 10-20 分鐘（含閱讀整理 + 設計排版） |
| 每週所需圖卡數量 | 5-10 張（假設每天 1-2 篇值得分享） |
| 每週花在圖卡的時間 | 1-3 小時 |
| 社群格式變體 | 每篇需 2-3 種尺寸（story + square + twitter） |
| 工具切換次數 | 閱讀器 → 筆記工具 → 設計工具 → 社群平台 = 4 次上下文切換 |

### 3.2 影響範圍

不解決此問題的直接後果：

1. **時間成本**: 每週 1-3 小時花在低價值的重複性排版工作
2. **內容損失**: 因為製作成本高，許多好文章被「讀了但沒分享」
3. **品牌損失**: 社群發文頻率降低或品質不一致，影響個人專業形象
4. **自動化斷裂**: 內容工作流中「閱讀 → 分享」這一環無法被程式化，成為管線瓶頸

### 3.3 現有解決方案的不足

| 解決方案 | 不足之處 |
|---------|---------|
| **Canva** | GUI 操作，無法嵌入自動化流程；需手動輸入文字；月費 $12.99+；模板通用但不針對「文章摘要」優化 |
| **Figma** | 定位是設計協作工具，非內容自動化；學習曲線高；無 AI 摘要 |
| **手動截圖 + 標記** | 品質不穩定，無法保持視覺一致性，無自動化能力 |
| **ChatGPT/Claude 生成文字 + 手動設計** | 摘要和設計是斷裂的兩步，仍需手動排版 |
| **AI 圖片生成（DALL-E、Midjourney）** | 適合藝術圖片，不適合文字密集的資訊圖卡；文字渲染不準確 |
| **自己寫 script** | 每次都要從零開始；沒有模板系統；缺乏多格式、批次、發布等完整管線 |

### 3.4 imgGen 如何解決這些問題

imgGen 的解法是**將「閱讀 → 摘要 → 設計 → 發布」整條管線程式化**：

| 痛點 | imgGen 解法 | 技術實現 |
|------|------------|---------|
| 手動整理文章重點 | AI 自動摘要，可選 provider | `src/extractor.py` — Claude/Gemini/GPT 多 provider，ExtractionConfig 調控語言/語氣/字數 |
| 手動設計排版 | 28 種預設主題 + article 模式 | `templates/` — Jinja2 HTML + CSS variables，統一設計系統（字體比例、動畫、版面） |
| 不同平台尺寸 | 4 種格式一鍵切換，可多格式同時輸出 | `src/screenshotter.py` — Playwright 截圖，430px 基準寬度 + format 控制高度 |
| 無法自動化 | CLI + config + preset + batch + watch | `main.py` Click CLI，TOML 三層 config，asyncio 批次，watchdog 監控 |
| 從 URL 到圖卡步驟太多 | 一行指令端到端 | `src/pipeline.py` — extract -> render -> capture 三階段管線 |
| 需要同時產出文案 | 自動生成平台文案 | `src/caption.py` — 針對 Twitter/LinkedIn/Instagram 各平台規則生成 |
| 發布需要手動操作 | 支援一鍵發推 | `src/publisher.py` — tweepy 整合 |
| 缺乏歷史追溯 | SQLite 歷史紀錄 + 統計 | `src/history.py` — WAL 模式，支援搜尋、統計、視覺化 |

**效率提升估算**：

| 工作項目 | 手動方式 | imgGen | 節省 |
|---------|---------|--------|------|
| 單篇文章 → 圖卡 | 10-20 分鐘 | < 30 秒 | 95%+ |
| 10 篇批次處理 | 2-3 小時 | < 60 秒（3 workers） | 99%+ |
| 多格式輸出（3 尺寸） | 30-60 分鐘 | 一次 AI 呼叫 + 3 張截圖 | 95%+ |
| 圖卡 + 社群文案 | 額外 5-10 分鐘/篇 | `--caption` 同步生成 | 90%+ |

## 4. 產品生命週期分析 (Product Lifecycle)

### 4.1 當前階段評估

基於已實現功能的盤點，imgGen 目前處於**成熟 MVP 向 Growth 過渡**的階段。

**已完成的能力矩陣**：

| 能力層 | 狀態 | 細節 |
|--------|------|------|
| 核心管線（Extract → Render → Screenshot） | 完全成熟 | 多 provider、多模板、多格式、多語言 |
| 輸入多樣性 | 完全成熟 | text / file / URL / batch，trafilatura 提取 |
| 模板系統 | 豐富 | 28 種主題覆蓋 19 種內容類型 + article 模式，統一設計系統 |
| 自動化基礎設施 | 完整 | config / preset / batch / watch / history |
| 社群發布 | 部分完成 | 文案生成（3 平台）+ 發布（僅 Twitter） |
| Web UI | 已上線 | FastAPI + React/TypeScript/Vite/TailwindCSS |
| 數據/歷史 | 完整 | SQLite 紀錄、搜尋、統計、視覺化 |
| 內容彙整 | 已實現 | digest 子命令，AI 合成週報 |

**階段判定依據**：
- 核心功能已超越 MVP：28 種主題遠超「最小可行」範圍
- 自動化能力（batch/watch/preset）顯示已進入「效率優化」階段
- Web UI 的加入代表從「CLI-only 工具」向「多介面產品」演進
- 社群發布功能（目前僅 Twitter）是 Growth 階段的早期信號

### 4.2 Growth 階段建議

imgGen 的 Growth 不是追求用戶數（這是個人工具），而是追求**使用密度**和**管線完整度**——讓工具在每天的內容工作流中更不可或缺。

**Growth 三大方向**：

#### 方向一：內容輸入管線的自動化深度

| 建議 | 理由 | 優先級 | 預估工作量 |
|------|------|--------|-----------|
| 整合 Readwise/Pocket API 自動匯入收藏文章 | 消除「手動複製 URL」的最後一步摩擦 | 高 | M |
| RSS feed 訂閱 → 自動觸發圖卡生成 | 結合 watch 機制，實現完全無人值守的內容管線 | 中 | M |
| 支援 Threads.net / Instagram 貼文 URL 解析（改善） | 社群內容二次加工場景 | 中 | S |

#### 方向二：輸出管線的多平台覆蓋

| 建議 | 理由 | 優先級 | 預估工作量 |
|------|------|--------|-----------|
| LinkedIn 直接發布 | 第二大目標平台，目前僅有文案生成 | 高 | M |
| Buffer/Hootsuite API 整合 | 支援排程發布，比直接發布更符合實際工作流 | 中 | M |
| Notion/Obsidian 自動存檔 | 將生成紀錄回寫到知識管理工具 | 低 | S |

#### 方向三：內容智慧化

| 建議 | 理由 | 優先級 | 預估工作量 |
|------|------|--------|-----------|
| AI 自動選擇最佳主題（基於內容語義） | 目前 AI 建議主題但準確度未知，可強化為自動匹配 | 高 | S |
| 生成結果 A/B 對比（同一篇用不同主題） | 幫助使用者選出最佳版本 | 中 | S |
| 社群 engagement 數據回饋（Twitter Analytics API） | 知道哪些主題/格式表現好，形成閉環 | 低 | L |

| 排程發布系統 | 定時發布比即時發布更符合內容策略 | 多平台發布穩定後 |
| 模板自訂系統 | 使用者可能想創建自己的視覺風格 | 收集到具體的自訂需求後 |

### 4.4 關鍵指標 (KPIs) 建議

因為 imgGen 是個人工具，KPI 設計不追蹤傳統 SaaS 指標（MAU/MRR），而是追蹤**工具效能**和**使用者價值感知**。

#### 效能指標（自動化可量測）

| 指標 | 定義 | 目標 | 量測方式 |
|------|------|------|---------|
| 端到端成功率 | 無 exception 完成的比例 | >= 95% | 日誌/history 統計 |
| 單張生成時間 | 含 API 呼叫的平均秒數 | < 30 秒 | pipeline 計時 |
| 批次吞吐量 | 10 篇 / 3 workers 總耗時 | < 60 秒 | batch_report.json |
| 模板覆蓋率 | 支援的內容類型占比 | 19/23 種（82%） | 手動盤點 |
| 格式覆蓋率 | 支援的社群平台尺寸 | 4 種主流格式 | 手動盤點 |

#### 使用密度指標（從 history.db 提取）

| 指標 | 定義 | 健康基準 | 量測方式 |
|------|------|---------|---------|
| 每週生成量 | 每週產出的圖卡數 | >= 10 張/週 | `history stats --days 7` |
| 主題使用分佈 | 不同主題的使用比例 | 至少 5 種主題被使用 | `history stats` |
| 批次使用頻率 | batch 模式占總使用的比例 | > 30% | history 記錄 |
| Web UI vs CLI 比例 | 兩介面的使用分佈 | 視偏好而定 | API 日誌 |

#### 品質指標（主觀 + 間接）

| 指標 | 定義 | 目標 | 量測方式 |
|------|------|------|---------|
| 直接發布率 | 生成圖卡後未修改直接發布的比例 | > 80% | 自我追蹤 |
| 主題建議採納率 | AI 建議主題被使用者接受的比例 | > 60% | history 中 theme vs theme_suggestion 比對 |
| 圖卡社群表現 | 圖卡貼文 vs 純文字貼文的 engagement 差異 | 圖卡 engagement > 2x | 平台分析 |
