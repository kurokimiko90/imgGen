# imgGen 產品路線圖 — Cycle 3
**狀態**: 草稿
**作者**: Alex (PM)
**最後更新**: 2026-03-28
**基準版本**: v1.6
**適用對象**: 獨立創作者，個人社群媒體發布用途

---

## 北極星指標

**每次執行從文章到可發布圖片的端到端時間 ≤ 15 秒，輸出品質令創作者無需二次修圖。**

這個指標同時兼顧速度（工具應該比手動做快）與品質（輸出應該直接可用），兩者缺一不可。

## 版本路線圖 v1.7 → v2.6

---

### v1.7 — 內容萃取升級（Better Extraction）

**一句話描述**: 以 Readability 演算法取代正規表達式 HTML 剝除，讓 URL 輸入的文章內容更乾淨、更完整。

**主要功能**:
- 整合 `trafilatura` 或 `newspaper4k` 做主要內文萃取
- 自動偵測並移除導覽列、頁腳、廣告、Cookie 提示等雜訊
- 保留文章標題、副標題、作者欄位供 AI 摘要參考
- 萃取失敗時自動降級回現有的 regex 剝除邏輯，不中斷流程
- 新增 `--raw` 旗標，可輸出萃取後的純文字供除錯檢查

**複雜度**: S

**Why now**: URL 輸入是最常用的輸入模式，但現有 regex 剝除對現代 SPA 網站幾乎失效，導致 AI 收到大量無意義的 JS 片段。這是最低成本、最高回報的改善。

---

### v1.8 — 浮水印與個人品牌（Watermark & Branding）

**一句話描述**: 允許在生成圖片上疊加 Logo、帳號名稱或自訂文字水印，強化社群媒體上的個人辨識度。

**主要功能**:
- `--watermark PATH` 支援 PNG/SVG Logo 疊加，位置可選（四角 + 中央）
- `--brand-name TEXT` 直接渲染文字水印，支援自訂字型大小與透明度
- 水印設定可寫入 `~/.imggenrc`，設定一次永久生效
- 支援 `--watermark-opacity 0.0-1.0` 調整透明度
- 預設不啟用，需明確設定才會出現（避免破壞現有輸出）

**複雜度**: S

**Why now**: 對社群創作者來說，品牌一致性是從「個人工具」升級為「個人媒體」的關鍵一步。實作上只需在 Playwright 截圖前在 HTML 模板注入額外的 CSS 疊加層，無需引入新依賴。

---

### v1.9 — 批次處理（Batch Processing）

**一句話描述**: 一次指令處理多個 URL、多個文字檔，或整個目錄，適合內容策展與每日發布流程。

**主要功能**:
- `--batch FILE` 接受純文字清單（每行一個 URL 或檔案路徑）
- `--batch-dir DIR` 對目錄內所有 `.txt` 檔執行批次生成
- 並行處理（可設定 `--workers N`，預設 3），縮短總等待時間
- 每筆輸出自動以來源 slug 命名，避免覆蓋
- 執行完成後產生 `batch_summary.json`，記錄每筆成功/失敗狀態與輸出路徑

**複雜度**: M

**Why now**: 單筆處理足夠應付偶爾使用，但一旦進入「每天發 3 則內容」的節奏，手動執行多次指令就變成摩擦力。批次處理讓工具從「偶爾用」升級為「工作流程核心」。

---

### v2.0 — 自訂 Prompt 樣式系統（Prompt Style Engine）

**一句話描述**: 讓使用者定義並儲存具名的 Prompt 風格（例如「科技評論」、「書摘」、「每日新聞」），徹底客製化 AI 摘要的語氣與結構。

**主要功能**:
- `~/.imggenrc` 新增 `[styles]` 區塊，可定義任意具名 Prompt 模板
- `--style NAME` 旗標套用已儲存的風格
- 內建 3 個預設風格：`default`（現有行為）、`punchy`（短句衝擊型）、`academic`（正式摘要型）
- Prompt 模板支援變數佔位符：`{title}`、`{content}`、`{word_limit}`
- `imggen styles list` 子指令列出所有可用風格與 Prompt 預覽

**複雜度**: M

**Why now**: 固定 Prompt 是目前最大的靈活性瓶頸。同一個人發布技術文章和生活感悟，需要的語氣截然不同。這功能讓工具真正「認識」使用者的聲音，而不是每次都輸出通用摘要。版號升至 v2.0 反映這是工具定位的根本性轉變。

---

### v2.1 — 監看模式（Watch Mode）

**一句話描述**: 監看指定目錄，當有新的 `.txt` 或 `.md` 文章檔案出現時自動觸發圖片生成，實現零摩擦的發布流程。

**主要功能**:
- `imggen watch DIR` 啟動監看守護程式（使用 `watchdog` 函式庫）
- 支援設定過濾規則：副檔名白名單、最小檔案大小
- 新檔出現後等待 2 秒確認寫入完成再觸發（避免編輯器存檔中途誤觸發）
- 生成完成後在終端機顯示通知，macOS 另外發送系統通知
- `--watch-style` 與 `--watch-theme` 可為 watch mode 設定獨立的預設值

**複雜度**: M

**Why now**: 對於使用 Obsidian、iA Writer 等本地寫作工具的創作者，watch mode 可以讓「儲存文章」與「產出社群圖片」之間的間隔縮短到接近零。這是工具從主動操作轉為被動整合的關鍵功能。

---

### v2.2 — 歷史紀錄與圖庫（History & Gallery）

**一句話描述**: 自動記錄每次生成的 metadata，並提供 `imggen gallery` 指令在本地瀏覽器中開啟縮圖圖庫。

**主要功能**:
- 每次生成自動寫入 `~/.imggen_history.json`：時間戳、來源、主題、格式、輸出路徑、摘要文字
- `imggen history` 指令列出最近 N 筆紀錄（支援 `--last N` 與 `--filter theme/format`）
- `imggen gallery` 在本地啟動靜態 HTML 頁面，展示所有輸出縮圖與 metadata
- `imggen history re-run ID` 以相同參數重新生成指定紀錄（適合主題更換後重出）
- 歷史檔案體積控制：預設保留最近 500 筆，可在 config 調整

**複雜度**: M

**Why now**: 隨著使用量增加，「上週那張深色主題的科技文章圖片在哪？」這類尋找需求必然出現。歷史紀錄功能讓工具從「用完即忘」進化為「可回溯的個人圖像資料庫」。

---

### v2.3 — 進階輸出品質控制（Output Quality Control）

**一句話描述**: 提供細粒度的圖片品質參數，包含解析度、壓縮率、顏色設定檔，滿足不同平台的上傳規格要求。

**主要功能**:
- `--dpi N` 設定輸出解析度（預設 72，印刷建議 300）
- `--quality 1-100` 控制 WebP/JPEG 壓縮品質（現有 WebP 輸出使用固定品質）
- `--color-profile srgb/p3` 選擇色彩設定檔（針對 iPhone/iPad 廣色域螢幕）
- `--max-size-kb N` 自動二分法調整壓縮率，確保輸出不超過指定檔案大小（適合有上傳限制的平台）
- 輸出 metadata 中記錄實際解析度與檔案大小

**複雜度**: S

**Why now**: Instagram、LinkedIn、Twitter 各有不同的建議圖片規格，目前輸出在某些平台會因為檔案過大被壓縮而降質。這個功能讓工具輸出真正符合平台規格，省去每次都要開 Preview.app 調整的步驟。

---

### v2.4 — 多卡片敘事模式（Multi-Card Story Mode）

**一句話描述**: 將長文章分解為 3-10 張序列圖卡，適合 Instagram 輪播或 Twitter Thread 格式的連續內容發布。

**主要功能**:
- `--multi N` 將文章拆分為 N 張圖卡，AI 負責拆分邏輯與各卡內容
- 第一張自動生成「封面卡」（標題為主），最後一張生成「結語卡」（CTA 或出處）
- 每張卡自動加上序號標示（如 1/5、2/5）
- 輸出為帶序號的檔案組（`output_01.png`、`output_02.png`...）
- `--multi-style carousel/thread` 分別針對 IG 輪播與 Twitter Thread 最佳化版面比例

**複雜度**: L

**Why now**: 長文單張圖卡壓縮資訊量有限，閱讀體驗差。多卡片格式是目前社群平台上表現最好的長內容格式（IG 輪播平均互動率是單張貼文的 3 倍）。這是從「摘要工具」升級為「社群內容引擎」的功能。

---

### v2.5 — 自訂 HTML/CSS 模板（Custom Template Engine）

**一句話描述**: 允許使用者提供自己的 HTML/CSS 模板檔案，突破內建三種主題的限制，實現完全客製化的視覺風格。

**主要功能**:
- `--template PATH` 指定自訂 HTML 模板（支援 Jinja2 語法）
- 模板可用的變數：`{{ title }}`、`{{ summary }}`、`{{ source }}`、`{{ date }}`、`{{ brand }}`
- `imggen template init` 輸出現有主題的 HTML/CSS 原始碼作為客製化起點
- 模板驗證：執行前檢查必要變數是否存在，缺少時給出明確錯誤訊息
- 模板路徑可存入 `~/.imggenrc` 設為永久預設

**複雜度**: M

**Why now**: 內建三種主題提供了良好的起點，但對於有強烈個人品牌需求的創作者（特定字型、配色、版型），任何固定主題都是上限而非支援。開放模板系統讓工具從「夠用」升格為「完美符合品牌」。

---

### v2.6 — 發布整合（Publish Integration）

**一句話描述**: 生成完成後，可選擇性地直接透過 API 上傳圖片到 Buffer、Cloudinary 或本地 Obsidian vault，消除最後一哩的手動步驟。

**主要功能**:
- `--publish buffer` 呼叫 Buffer API 將圖片加入排程佇列（需設定 `BUFFER_API_KEY`）
- `--publish cloudinary` 上傳到 Cloudinary 並回傳 CDN URL（適合部落格內嵌用途）
- `--publish obsidian` 複製到指定的 Obsidian vault 附件目錄並插入今日日記
- 所有發布設定透過 `~/.imggenrc` 的 `[publish]` 區塊管理
- 發布失敗不影響本地圖片輸出，錯誤訊息清楚說明原因與排查步驟

**複雜度**: L

**Why now**: 生成圖片後還要手動打開各平台上傳，是整個流程最後也最煩人的一步。發布整合將工具從「生成工具」蛻變為「完整發布管線」，讓創作者真正做到零切換發布。

## 功能選擇矩陣

評分標準說明:
- **用戶影響 (1-5)**: 對創作者日常工作流程的實質改善程度
- **可行性 (1-5)**: 技術實作難度低 = 高分（5 = 一週內可完成，1 = 需要數月重構）
- **WOW Factor (1-5)**: 使用者第一次看到時的驚喜感與分享意願

| 候選功能 | 用戶影響 | 可行性 | WOW Factor | 總分 |
|----------|----------|--------|------------|------|
| 自訂 Prompt 樣式系統 (v2.0) | 5 | 4 | 4 | **80** |
| 浮水印與個人品牌 (v1.8) | 4 | 5 | 4 | **80** |
| 批次處理 (v1.9) | 4 | 4 | 3 | **48** |
| 多卡片敘事模式 (v2.4) | 5 | 2 | 5 | **50** |
| 內容萃取升級 (v1.7) | 4 | 5 | 2 | **40** |

> 計算方式: 用戶影響 × 可行性 × WOW Factor = 總分

---

### 獲勝功能分析

**並列冠軍: 自訂 Prompt 樣式系統 (v2.0) 與 浮水印與個人品牌 (v1.8)，兩者並列 80 分。**

兩者雖然總分相同，但切入面不同，建議依序執行：

**首先實作 v1.8 浮水印（可行性 5，一週內可交付）**

浮水印的可行性最高，實作成本極低（HTML 模板注入一個絕對定位的 `<img>` 或 `<div>`），但對社群媒體創作者的品牌曝光價值極高。每一張圖片發布出去都帶著你的 Logo，這是複利效應最強的功能。它幾乎不需要新增依賴、不改變核心流程、不需要用戶學習任何新語法——只要在 `~/.imggenrc` 設定一次，往後每張圖自動帶品牌。

**其次實作 v2.0 自訂 Prompt 樣式（用戶影響 5，是工具定位的轉折點）**

Prompt 樣式系統的用戶影響同樣是最高分，且它解決的是目前最根本的限制：固定的 AI 輸出語氣讓工具無法真正反映創作者的個人聲音。對於同時經營技術部落格和生活圖文的人來說，這是最明顯的痛點。技術上只需擴充 config 解析邏輯與 prompt 組裝模組，可行性給 4 分（需要設計良好的模板變數系統，但不涉及 Playwright 或 AI SDK 的大改動）。

**多卡片敘事模式 (v2.4) 雖然 WOW Factor 最高，但可行性僅 2 分——需要重新設計 Playwright 截圖流程、AI Prompt 拆分邏輯、以及輸出命名系統，複雜度是其他功能的 3 倍以上，不適合作為優先投入的起點。待 v2.0 的 Prompt 系統穩定後再推進，可以直接複用其模板基礎設施。**

## 版本依賴圖

```
v1.6 (基準)
  └── v1.7 內容萃取升級       ← 獨立，隨時可做
  └── v1.8 浮水印品牌         ← 獨立，建議優先
  └── v1.9 批次處理           ← 依賴 v1.7（更乾淨的內容萃取）
        └── v2.0 Prompt 樣式  ← 獨立，但受益於 v1.9 的批次驗證
              └── v2.1 Watch Mode   ← 依賴 v1.9 架構
              └── v2.3 品質控制     ← 獨立
              └── v2.4 多卡片       ← 依賴 v2.0 Prompt 系統
                    └── v2.5 自訂模板 ← 依賴 v2.4 多卡片框架
                    └── v2.2 歷史圖庫 ← 獨立，隨時可插入
                          └── v2.6 發布整合 ← 依賴 v2.2 歷史紀錄
```

## 明確不做的事（本 Cycle）

| 需求 | 不做的理由 | 重新評估條件 |
|------|-----------|------------|
| 雲端同步 / 多裝置 | 單人工具，本地即是唯一需求，雲端引入安全與成本複雜度 | 使用者明確有多台機器工作流程需求 |
| Web UI / GUI | CLI 是核心優勢，GUI 增加維護負擔且不符合工作流自動化需求 | 使用者量超過 1 人（目前為 0） |
| 社群帳號認證管理 | OAuth 流程複雜，安全風險高，與工具定位無關 | 考慮 v2.6 之後的獨立安全審查 |
| 影片/GIF 輸出 | Playwright 截圖管線不適合動態輸出，需要完全不同的技術棧 | 確認有具體使用場景後再評估 |
| 多語言界面 | 單人使用，沒有本地化需求 | 永不 |
